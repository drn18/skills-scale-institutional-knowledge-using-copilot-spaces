## Purpose
Ensure Technical Writers have all necessary information to create accurate, comprehensive documentation for new features and updates.

## When to Use
Use this checklist when:
- New features are being developed
- Existing features are being significantly updated
- API changes are being made
- Process documentation needs updating

## Pre-handoff from Development

### Feature Information
- [ ] Feature overview and purpose documented
- [ ] User stories or use cases provided
- [ ] Target audience identified (end users, developers, admins, etc.)
- [ ] Feature flag or rollout plan communicated

### Technical Details
- [ ] API endpoints and methods documented (if applicable)
- [ ] Configuration options and defaults listed
- [ ] Dependencies and prerequisites identified
- [ ] Known limitations or constraints documented
- [ ] Error messages and troubleshooting guidance provided

### Access and Resources
- [ ] Access to staging/test environment provided
- [ ] Test accounts or sample data available
- [ ] Technical contact person identified for questions
- [ ] Links to relevant PRs, design docs, or technical specs

## Documentation Requirements

### Content to Create or Update
- [ ] User guide or how-to documentation
- [ ] API reference documentation (if applicable)
- [ ] Configuration guide
- [ ] Troubleshooting section
- [ ] Migration guide (for breaking changes)
- [ ] Release notes entry
- [ ] Internal runbook or operations guide (if applicable)

### Documentation Standards
- [ ] Follow existing documentation style guide
- [ ] Include code examples and screenshots where helpful
- [ ] Ensure accessibility (alt text, clear language, proper heading structure)
- [ ] Add version information and last updated date
- [ ] Include links to related documentation

### Review Process
- [ ] Technical accuracy review by developer or subject matter expert
- [ ] Peer review by another Technical Writer (if available)
- [ ] Product Manager review for messaging and completeness
- [ ] UX review for screenshots and UI terminology consistency

## Documentation Handoff Timing

### Early Involvement (Recommended)
- [ ] Technical Writer involved during planning phase
- [ ] Documentation requirements included in Definition of Done
- [ ] Draft documentation created during development
- [ ] Documentation reviewed before feature reaches QA

### Release Preparation
- [ ] Final documentation updates completed before release
- [ ] Release notes drafted and reviewed
- [ ] Documentation published or scheduled for release day
- [ ] Support team notified of documentation updates

## Post-release Activities
- [ ] Monitor support channels for documentation gaps
- [ ] Gather feedback from users and support team
- [ ] Create follow-up documentation tasks as needed
- [ ] Update documentation based on user feedback
